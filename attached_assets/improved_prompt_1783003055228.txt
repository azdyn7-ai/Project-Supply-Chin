**PHASE 1: THE SECURE BUILD PIPELINE (SLSA Level 3 + SBOM + Signing)**

**Step 1: The Microservice Application**
* Create a Go web application (NOT just "Hello World") that:
  - Has a `/health` endpoint returning JSON status
  - Has a `/version` endpoint returning build info
  - Has 3+ third-party dependencies (e.g., gin-gonic, go-uuid, logrus) — these are needed for meaningful SBOM analysis
  - Includes a `go.sum` file for reproducible builds
* Provide the full `main.go`, `go.mod`, `go.sum`, and `Dockerfile` (multi-stage build).

**Step 2: The Build Pipeline Script**
* Write `build_pipeline.sh` that implements a SLSA Level 3 compliant build:
  1. **Build** the Docker image from the Dockerfile (hermetic, no network access during build)
  2. **Generate SLSA Provenance** — this is CRITICAL and must NOT be skipped:
     - Use `slsa-github-generator` OR manually create a SLSA v1.0 provenance JSON that includes: builder ID, build type, source URI, commit hash, build inputs, output image digest, and timestamp
     - Sign the provenance using Cosign: `cosign attest --predicate provenance.json --type slsaprovenance`
  3. **Generate SBOM** using `syft` in SPDX-JSON format
  4. **Attest SBOM** to the image using `cosign attach sbom`
  5. **Sign** the image using `cosign sign` with keyless mode (Fulcio + Rekor) OR a generated key pair if keyless is unavailable
  6. **Vulnerability Scan** the SBOM using `grype` and save the report
  7. **Push** image, signatures, attestations, and SBOM to the local registry
* The script must print a summary at the end showing: image digest, provenance status, SBOM package count, vulnerability counts (critical/high/medium/low), signature verification status.
* The script must exit with error code if any step fails.

**Step 3: Verification Script**
* Write `verify_artifacts.sh` that:
  1. Verifies the image signature: `cosign verify`
  2. Verifies the SLSA provenance attestation: `cosign verify-attestation --type slsaprovenance`
  3. Downloads and parses the SBOM: `cosign download sbom`
  4. Extracts the list of ALL binaries and packages from the SBOM and saves to `allowed_binaries.json` and `allowed_packages.json`
  5. Prints: "VERIFIED" if all checks pass, "FAILED" with specific reason if any check fails

**PHASE 3: PROVENANCE ENRICHMENT SERVICE (THE CORE INNOVATION)**

This is the MOST IMPORTANT phase. This is what makes this project novel — connecting build-time metadata to runtime monitoring.

**Step 5: The Enrichment Service**
* Write a Python or Go service (`enrichment_service/`) that:
  1. Watches for new Pod deployments via Kubernetes Watch API
  2. When a new Pod is deployed, extracts the image reference
  3. Fetches the SBOM for that image from the OCI registry using `cosign download sbom`
  4. Parses the SBOM and extracts: list of allowed binaries (from `/usr/bin`, `/bin`, `/usr/local/bin`), list of allowed packages, list of allowed network ports
  5. Fetches the SLSA provenance and extracts: builder identity, SLSA level, source repository
  6. Fetches the signature verification status
  7. Generates dynamic Falco rules for this specific image: a whitelist of allowed process names, a whitelist of allowed network connections
  8. Stores the enrichment data in a ConfigMap named `provenance-<image-hash>` in the same namespace
  9. Triggers Falco rules reload via `falcoctl` or API
* Provide the full Dockerfile for this service
* Provide the Kubernetes Deployment + Service + RBAC manifests
* Provide a MutatingWebhookConfiguration that calls this service on every Pod creation

**Step 6: SBOM-Aware Falco Rules**
* Write `falco_rules.local.yaml` with THREE categories of rules:
  1. **SBOM-Enriched Rules** (the novel contribution):
     - Rule: "Process execution not in SBOM" — fires when a process executes in a container AND that process binary is NOT in the `allowed_binaries.json` from the enrichment ConfigMap for that image
     - Rule: "Package not in SBOM" — fires when a new file appears in a package directory not listed in the SBOM
     - Rule: "Network connection not expected" — fires when an outbound connection is made to a port/IP not matching the SBOM-declared dependencies
     - Each alert MUST include enrichment context: image name, signer identity, SLSA level, SBOM source
  2. **Standard Runtime Rules** (baseline for comparison):
     - Shell spawned in container
     - Read/write to sensitive directories (/etc/shadow, /root)
     - Unexpected inbound network connections
  3. **Supply Chain Specific Rules**:
     - Image running with SLSA level below threshold
     - Image signed by unknown identity
     - ConfigMap modified after initial enrichment

**Step 10: Measurement & Metrics Collection**

* Write `collect_metrics.sh` that runs the FULL evaluation:

  **Detection Accuracy Metrics** (for each scenario, 10 runs):

  - False Negative Rate (FNR) = undetected attacks / total attacks
  - Detection Latency = time from attack execution to first alert (average, min, max, stddev)

  **Performance Overhead Metrics** (using Prometheus queries):

  - Network Overhead: bytes transferred with and without monitoring (from Prometheus `node_network_transmit_bytes_total`)

  - Mode A: No security (baseline) — uninstall Kyverno, Falco, Tetragon
  - Mode B: Build-time only — keep Kyverno admission, remove Falco/Tetragon/Enrichment
  - Mode C: Full integrated framework — all components active
